# ppt

`ppt` 是一个面向单个投资者的永久投资组合 CLI。它只做两件事：在 OSS 中保存有符号交易批次，以及把现有持仓、自动均衡和一笔新增人民币资金合并成一条可执行交易命令。

长期目标固定为四桶各 25%，决策优先级固定为：

1. 四大桶均衡；
2. 桶内标的等市值；
3. 全部资产的美元与人民币市值接近 50% / 50%。

后一级只能在前面各级同样好的方案之间选择，不能牺牲更高优先级。项目不使用风险平价，也不会根据行情改变 25% 目标。

## 安装与环境

需要 Python 3.11+、[uv](https://docs.astral.sh/uv/) 和已完成认证的 `ossutil`：

```bash
uv sync --locked --extra dev
export PP_OSS_BUCKET=your-private-bucket
uv run ppt --help
```

`PP_OSS_BUCKET` 必填，避免在环境配置缺失时误读或覆盖某个默认账本。可用 `OSSUTIL_PATH` 指定 `ossutil` 可执行文件，默认值为 `ossutil`。程序不会自动读取 `.env` 文件。

OSS 身份需要当前对象与备份的读写权限，以及租约锁使用的 `oss:GetObject`、`oss:PutObject`、`oss:DeleteObject` 和 `oss:ListObjects`；权限不足会明确失败，不会被当作“账本不存在”。

正常运行时，行情来自 Yahoo Finance。需要复现固定行情或隔离验收时，可把 `PP_PRICE_FILE` 指向一个 JSON 文件：

```json
{
  "prices": {
    "SPYM": 75.0,
    "AVUV": 95.0,
    "VGIT": 58.0,
    "GLDM": 60.0,
    "518880.SS": 6.0,
    "SGOV": 100.0,
    "511360.SS": 100.0
  },
  "usdcny": 7.2,
  "history": {
    "SPYM": [70.0, 71.0, 72.0]
  }
}
```

`history` 可省略；缺少历史只会显示“数据不足”，不会改变交易方案。当前七个价格或 USD/CNY 缺失、非数值、非有限或不大于零时，命令失败且不会写账本。

## 命令

帮助中只有四个子命令：

```text
init
buy
plan
history
```

### `ppt init`

经确认后重置持仓和全部交易历史：

```bash
uv run ppt init
```

非交互环境可使用 `uv run ppt init --yes`。如果 OSS 中已有对象，程序先在 `backups/` 下创建不可覆盖的唯一备份，再写入空账本；如果没有旧对象，则直接创建。旧对象即使不是当前账本格式，也会先原样备份后再重置。

### `ppt buy`

一次原子记录一个或多个标的：

```bash
uv run ppt buy 'SPYM#10@75.20' '518880.SS#200@5.83' 'SGOV#-3@100.40'
```

格式为 `代码#有符号股数@成交价`：

- 正股数是买入，负股数是卖出，零不合法；
- 美股按整股，A 股 ETF 按 100 份；
- 成交价使用标的原币；USD 交易按记录当时的有效 USD/CNY 折算；
- 同一批次中每个代码只能出现一次；
- 任一参数无效、卖出后任一持仓为负、备份失败、锁失败或写入失败时，整批不写入。

每个批次只保存交易事实和历史汇率；当前持仓与现金流由全部批次回放得出，不在账本中保存第二份可漂移状态。

### `ppt plan <人民币金额>`

`plan` 读取账本和行情，生成自动均衡与新增资金合并后的净交易：

```bash
uv run ppt plan 10000
```

金额必须是正的有限人民币数值。规划规则如下：

1. 交易前任一桶低于 15% 或高于 35% 时，开放存量跨桶卖出，并以 25% 为目标自动均衡；否则不允许桶净流出，只用新增资金优先填补较低配桶。
2. 多标的桶以等市值为目标。新增买入优先较低市值标的；某标的占桶市值超过 60% 时，才允许存量桶内换仓。黄金和现金的主动换仓方向为 `GLDM → 518880.SS`、`SGOV → 511360.SS`。
3. 只有四桶与桶内偏差都不变差时，才选择更接近 USD/CNY 50% / 50% 的方案。由于股票和债券只有美元标的，50% / 50% 可能无法达到，这是优先级约束的正常结果。
4. 最终持仓必须符合交易单位和非负约束；买入支出不能超过新增金额与卖出收入之和。

规划器直接选择最终持仓，再用“最终持仓减当前持仓”生成交易，所以存量均衡、桶内均衡、币种均衡、两种换仓和新增买入天然合并为一个净额。同一标的最多出现一次，零净额被删除，不会出现同一标的既买又卖。

输出包括：

- 每个标的的有符号净股数和预计人民币金额；
- 新增金额、卖出收入、买入支出和未使用金额；
- 买入前后四桶、桶内、币种三级偏差；
- 近期趋势和相关性异常提示；
- 一条与净交易完全一致、可复制执行的 `ppt buy ...` 命令。

趋势与相关性只生成提示，绝不进入规划评分，也不改变 25% 目标、15%–35% 走廊或三级优先级。`plan` 不写 OSS 账本。

### `ppt history`

```bash
uv run ppt history
```

顶部展示累计投入、累计取出、当前市值、收益额和收益率；随后按批次倒序展示时间、标的、有符号股数与原币成交价。

一批交易的净人民币现金流为所有有符号成交额之和。正净额计入累计投入，负净额的绝对值计入累计取出，因此一次内部换仓不会把买卖毛额同时虚增为外部资金流：

```text
收益额 = 当前市值 + 累计取出 - 累计投入
收益率 = 收益额 / 累计投入
```

累计投入为零时，收益率显示“不可计算”。`history` 只读账本；为计算当前市值会读取有效行情。

## 固定资产与估值

| 桶 | 标的 | 币种 | 交易单位 |
|---|---|---:|---:|
| 股票 | SPYM、AVUV | USD | 1 股 |
| 债券 | VGIT | USD | 1 股 |
| 黄金 | GLDM | USD | 1 股 |
| 黄金 | 518880.SS | CNY | 100 份 |
| 现金 | SGOV | USD | 1 股 |
| 现金 | 511360.SS | CNY | 100 份 |

所有组合市值、预算约束和偏差统一按人民币计算。用户不能增加标的或修改目标比例。

## 账本与失败语义

OSS 中只有一个当前账本对象，结构为：

```json
{
  "initialized_at": "2026-07-22T12:00:00+08:00",
  "batches": [
    {
      "id": "uuid",
      "executed_at": "2026-07-22T12:01:00+08:00",
      "usdcny": 7.2,
      "trades": [
        {"ticker": "SPYM", "shares": 10, "price": 75.2}
      ]
    }
  ]
}
```

除 `init` 外，旧格式或结构不一致的对象会被拒绝，不迁移、不兼容。每次覆盖当前账本前都必须在持有 OSS 串行锁时完成唯一备份；备份、锁或写入任一步失败都不会继续后续写入。OSS 单对象替换负责原子可见性。每个锁持有者使用独立的限时租约对象：正常结束和业务异常都会删除自己的租约，释放调用本身失败时命令报错且租约可到期回收，不会删除其他持有者的锁。

只有 `init` 和 `buy` 修改账本。项目不提供独立 `sell`、`status`、`undo`、`rebalance`、`config`、清理命令、券商连接、自动下单、GUI、Web 服务或多用户能力。

## 开发与验证

代码按职责分层：纯计算不读环境、不做 IO、不输出；OSS 与行情模块只处理 IO；`ppt.display` 统一组装终端组件；CLI 只解析参数和编排。

```bash
uv run pytest
uv run ruff check .
```

交付验收还会在隔离 OSS 和固定行情中真实执行 `init → buy → plan → 执行 plan 输出命令 → history`，并验证重启持久化、走廊、三级优先级、预算、超卖、并发锁、备份失败与无残留锁。
