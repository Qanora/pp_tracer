# ppt

`ppt` 是一个面向单个投资者的永久投资组合 CLI。它在 OSS 中保存有符号交易批次，展示组合当前配置，并把现有持仓、自动均衡和一笔新增人民币资金合并成一条可执行交易命令。

长期目标固定为四桶各 25%，决策优先级固定为：

1. 四大桶均衡；
2. 桶内标的等市值；
3. 全部资产的美元与人民币市值接近 50% / 50%。

后一级只能在前面各级同样好的方案之间选择，不能牺牲更高优先级。项目不使用风险平价，也不会根据行情改变 25% 目标。

## 安装与环境

需要 Python 3.11+、[uv](https://docs.astral.sh/uv/) 和已完成认证的 `ossutil`：

```bash
uv sync --locked --extra dev
export PP_OSS_BUCKET=your-private-bucket
uv run ppt --help
```

`PP_OSS_BUCKET` 必填，避免在环境配置缺失时误读或覆盖某个默认账本。可用 `OSSUTIL_PATH` 指定 `ossutil` 可执行文件，默认值为 `ossutil`。程序不会自动读取 `.env` 文件。

OSS 身份需要当前对象与备份的读写权限，以及租约锁使用的 `oss:GetObject`、`oss:PutObject`、`oss:DeleteObject` 和 `oss:ListObjects`；权限不足会明确失败，不会被当作“账本不存在”。

正常运行时，行情来自 Yahoo Finance。需要复现固定行情或隔离验收时，可把 `PP_PRICE_FILE` 指向一个 JSON 文件：

```json
{
  "prices": {
    "SPYM": 75.0,
    "AVUV": 95.0,
    "VGIT": 58.0,
    "GLDM": 60.0,
    "518880.SS": 6.0,
    "SGOV": 100.0,
    "511360.SS": 100.0
  },
  "usdcny": 7.2,
  "history": {
    "SPYM": {
      "2026-07-20": 70.0,
      "2026-07-21": 71.0,
      "2026-07-22": 72.0
    }
  },
  "usdcny_history": {
    "2026-07-20": 7.18,
    "2026-07-21": 7.19,
    "2026-07-22": 7.2
  }
}
```

`history` 和 `usdcny_history` 可省略，只供 `status` 的趋势、相关性和最近 30 个共同有效交易日回测使用；键必须是 ISO 日期。缺少、无效或不足 30 个共同有效交易日的可选历史只会显示“数据不足”，不会改变交易方案。当前七个价格或 USD/CNY 缺失、非数值、非有限或不大于零时，命令失败且不会写账本。

## 命令

帮助中只有五个子命令：

```text
init
buy
status
plan
history
```

### `ppt init`

经确认后重置持仓和全部交易历史：

```bash
uv run ppt init
```

非交互环境可使用 `uv run ppt init --yes`。如果 OSS 中已有对象，程序先在 `backups/` 下创建不可覆盖的唯一备份，再写入空账本；如果没有旧对象，则直接创建。旧对象即使不是当前账本格式，也会先原样备份后再重置。

### `ppt buy`

一次原子记录一个或多个标的：

```bash
uv run ppt buy 'SPYM#10@75.20' '518880.SS#200@5.83' 'SGOV#-3@100.40'
```

格式为 `代码#有符号股数@成交价`：

- 正股数是买入，负股数是卖出，零不合法；
- 美股按整股，A 股 ETF 按 100 份；
- 成交价使用标的原币；USD 交易按记录当时的有效 USD/CNY 折算；
- 终端成交价按市场精度显示：美股固定 2 位小数，沪市 ETF 固定 3 位小数；
- 同一批次中每个代码只能出现一次；
- 任一参数无效、卖出后任一持仓为负、备份失败、锁失败或写入失败时，整批不写入。

每个批次只保存交易事实和历史汇率；当前持仓与现金流由全部批次回放得出，不在账本中保存第二份可漂移状态。

### `ppt status`

读取账本和当前行情，展示组合此刻的持仓与配置：

```bash
uv run ppt status
```

输出包括：

- 组合人民币总市值、累计投入、累计取出、净投入、累计盈亏、简单收益率、当前 USD/CNY，以及四桶是否越过 15%–35% 走廊；
- 七个固定标的的桶、币种、股数、原币现价、人民币市值、组合占比、桶内占比与桶内目标；
- 四桶的人民币市值、当前占比、固定 25% 目标、有符号偏差和走廊位置；
- 美元与人民币资产折合人民币后的市值、当前占比、固定 50% 目标和有符号偏差；
- 四桶、桶内、币种三级最大偏差；
- 当前实际持仓按历史 USD/CNY 折算的桶级人民币曲线所产生的近期趋势和相关性异常；
- 最近 30 个共同有效交易日当前持仓回测的当前回撤、最大回撤与最大涨幅。

累计盈亏为“当前市值 + 累计取出 - 累计投入”，简单收益率为“累计盈亏 / 累计投入”，不年化。回测将当前股数固定在最近 30 个共同有效交易日，用带日期的标的收盘价和历史 USD/CNY 折算每日人民币市值：当前回撤是最新市值相对窗口内此前最高市值的降幅，最新值创新高时为 0；最大回撤是峰值到其后低点的最大降幅；最大涨幅是低点到其后高点的最大累计涨幅，不是最大单日涨幅。因此这些指标衡量当前风险暴露，不是账户实际经历的表现。未记录的分红、费用和税费不计入；空持仓或不足 30 个共同有效交易日时三项指标均显示“不可计算”。

趋势与相关性同样基于当前实际持仓的桶级人民币市值曲线，不使用固定代表标的；相关性最多取最近 30 个共同有效交易日。空桶或所需标的、汇率历史不足时只显示“数据不足”，不会用其他标的代替。

空持仓仍展示全部七个标的和 ¥0 总市值，无法由零市值定义的占比显示为 `—`。`status` 不展示交易批次或交易建议，也不写账本、创建备份或取得写锁。

### `ppt plan <人民币金额>`

`plan` 读取账本和行情，生成自动均衡与新增资金合并后的净交易：

```bash
uv run ppt plan 10000
```

金额必须是正的有限人民币数值。规划规则如下：

1. 交易前任一桶低于 15% 或高于 35% 时，开放存量跨桶卖出，并以 25% 为目标自动均衡；否则不允许桶净流出，只用新增资金优先填补较低配桶。
2. 多标的桶以等市值为目标。新增买入优先较低市值标的；某标的占桶市值超过 60% 时，才允许存量桶内换仓。黄金和现金的主动换仓方向为 `GLDM → 518880.SS`、`SGOV → 511360.SS`。
3. 只有四桶与桶内偏差都不变差时，才选择更接近 USD/CNY 50% / 50% 的方案。由于股票和债券只有美元标的，50% / 50% 可能无法达到，这是优先级约束的正常结果。
4. 最终持仓必须符合交易单位和非负约束；买入支出不能超过新增金额与卖出收入之和。

规划器直接选择最终持仓，再用“最终持仓减当前持仓”生成交易，所以存量均衡、桶内均衡、币种均衡、两种换仓和新增买入天然合并为一个净额。同一标的最多出现一次，零净额被删除，不会出现同一标的既买又卖。

输出包括：

- 每个标的的有符号净股数和预计人民币金额；
- 新增金额、卖出收入、买入支出和未使用金额；
- 买入前后四桶、桶内、币种三级偏差；
- 一条与净交易完全一致、可复制执行的 `ppt buy ...` 命令。

行情原始精度仍用于估值、均衡评分和资金约束计算；表格以及生成命令中的价格按市场精度输出，美股固定 2 位小数，沪市 ETF 固定 3 位小数。

`plan` 只展示交易结果及其资金与均衡数据；趋势和相关性移至 `status`，绝不进入规划评分。`plan` 不写 OSS 账本。

### `ppt history`

```bash
uv run ppt history
```

顶部展示累计投入、累计取出和净投入；随后按批次倒序展示时间、标的、有符号股数与原币成交价。

一批交易的净人民币现金流为所有有符号成交额之和。正净额计入累计投入，负净额的绝对值计入累计取出，因此一次内部换仓不会把买卖毛额同时虚增为外部资金流。

`history` 只读账本，不读取行情；行情源不可用时仍可查看完整资金流和交易批次。当前市值、累计盈亏和收益率统一在 `status` 展示。

## 固定资产与估值

| 桶 | 标的 | 币种 | 交易单位 |
|---|---|---:|---:|
| 股票 | SPYM、AVUV | USD | 1 股 |
| 债券 | VGIT | USD | 1 股 |
| 黄金 | GLDM | USD | 1 股 |
| 黄金 | 518880.SS | CNY | 100 份 |
| 现金 | SGOV | USD | 1 股 |
| 现金 | 511360.SS | CNY | 100 份 |

所有组合市值、预算约束和偏差统一按人民币计算。用户不能增加标的或修改目标比例。

## 账本与失败语义

OSS 中只有一个当前账本对象，结构为：

```json
{
  "initialized_at": "2026-07-22T12:00:00+08:00",
  "batches": [
    {
      "id": "uuid",
      "executed_at": "2026-07-22T12:01:00+08:00",
      "usdcny": 7.2,
      "trades": [
        {"ticker": "SPYM", "shares": 10, "price": 75.2}
      ]
    }
  ]
}
```

除 `init` 外，旧格式或结构不一致的对象会被拒绝，不迁移、不兼容。每次覆盖当前账本前都必须在持有 OSS 串行锁时完成唯一备份；备份、锁或写入任一步失败都不会继续后续写入。OSS 单对象替换负责原子可见性。每个锁持有者使用独立的限时租约对象：正常结束和业务异常都会删除自己的租约，释放调用本身失败时命令报错且租约可到期回收，不会删除其他持有者的锁。

只有 `init` 和 `buy` 修改账本。项目不提供独立 `sell`、`undo`、`rebalance`、`config`、清理命令、券商连接、自动下单、GUI、Web 服务或多用户能力。

## 开发与验证

代码按职责分层：纯计算不读环境、不做 IO、不输出；OSS 与行情模块只处理 IO；`ppt.display` 统一组装终端组件；CLI 只解析参数和编排。

```bash
uv run pytest
uv run ruff check .
```

交付验收还会在隔离 OSS 和固定行情中真实执行 `init → buy → status → plan → 执行 plan 输出命令 → history`，并验证 `status` 只读、重启持久化、走廊、三级优先级、预算、超卖、并发锁、备份失败与无残留锁。
